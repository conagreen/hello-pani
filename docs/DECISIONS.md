## 0. 요구사항 분석

본격적인 의사결정에 앞서, 시스템이 만족해야 할 요구사항을 분석한다.
이후 쟁점들의 모든 트레이드오프는 이 요구사항을 기준으로 평가된다.

### 0.1 도메인 개요

> *00시에 오픈되는 한정 10개 상품의 선착순 예약 / 결제*

- **한정성** — 재고가 10개로 매우 적다. 거의 모든 사용자가 실패한다는 전제에서 시스템을 설계해야 한다.
- **시간성** — 오픈 시각이 정해져 있다. 트래픽은 평균적으로 분산되지 않고 짧은 시간에 첨두(spike) 형태로 몰린다.
- **분산성** — 앱 서버가 2대 이상이다. 정합성·공정성 메커니즘은 특정 앱 인스턴스의 인메모리 상태에 의존할 수 없다.
- **결제 통합** — 예약 확정과 결제가 단일 API(POST /booking) 경계 안에서 처리된다. 어느 한쪽이 실패했을 때의 보상 처리가 핵심 설계 포인트가 된다.

### 0.2 트래픽 / 부하 프로파일

| 구분 | 값 |
|---|---|
| 평시 | 50 TPS |
| 오픈 직후 1-5분 | 500-1,000 TPS |
| 5분간 누적 요청 | 약 150,000-300,000건 |
| 성공해야 하는 건수 | 정확히 10건 |
| 기대 성공률 | 약 0.003-0.007% |

이 *비대칭성*이 이 시스템 설계의 핵심이다.

> **이 시스템은 "성공시키는 시스템"이 아니라 "거절을 빠르고 공정하게 수행하는 시스템"이다.**

보통의 백엔드 설계는 *성공 경로*를 최적화하지만, 여기서는 *거절 경로*가 충분히 빠르고 명확한 응답을 주어야 한다. 이 관점이 이후 모든 쟁점에 영향을 준다.

### 0.3 기능적 요구의 3축

세 가지가 동시에 만족되어야 한다. 어느 하나라도 깨지면 사업적 신뢰가 깨진다.

1. **정합성 (correctness)** — 정상 운영 상태에서는 재고 10개가 모두 판매 가능해야 하며, 어떤 경우에도 초과 판매가 발생하지 않아야 한다. Redis 장애, PG 결과 불명, 보상 실패처럼 판매 가능 상태를 안전하게 판정할 수 없는 경우에는 DB로 우회하지 않고 fail-fast한다. 즉 장애 상황에서는 availability보다 correctness를 우선한다.
2. **공정성 (fairness)** — 모든 사용자에게 동등한 기회를 제공한다. 서버에 도달한 이후에는 운영자 편향이나 특정 앱 인스턴스 편향이 없어야 한다.
3. **멱등성 (idempotency)** — 짧은 간격의 중복 요청이 같은 결과를 낸다. 두 번 결제되지 않는다.

### 0.4 결제 도메인의 구조와 확장성 요구

- **지원 결제 수단**: 신용카드 / Y페이 / Y포인트
- **허용 조합**:
   - 카드 단독 / Y페이 단독 / 포인트 단독
   - 카드 + 포인트
   - Y페이 + 포인트
- **금지 조합**: 카드 + Y페이 (서로 다른 외부 결제망)
- **확장 요구**: 신규 결제 수단이 추가되어도 **Booking API의 비즈니스 로직 수정 최소화**

이 요구는 "확장점이 어디에 있어야 하는지"를 명시적으로 가리킨다.
따라서 결제 처리 로직을 외부에서 교체·추가 가능한 단위로 분리해야 한다는 신호로 읽는다.

### 0.5 장애 / 예외 시나리오

- **Redis 장애** — Fallback 전략 수립 및 근거 제시 요구
- **결제 실패** — 한도 초과 등. 부분 실패 시 전체 거래를 일관되게 무효화하거나, 재시도 / 보상으로 회복할 수 있어야 함

### 0.6 범위 밖 (Out of Scope)

명시적으로 제외된 영역. 이 문서와 코드는 이 영역에 대한 결정을 다루지 않는다.

- 회원 인증 / 로그인 보안
- 실제 PG사와의 연동 (인터페이스 / 어댑터로만 구성)

### 0.7 인프라 / 운영 제약

- **분산 환경**: 앱 서버 2대 이상.
- **증설 제한**: scale-up·scale-out이 제한적인 상황을 가정 → "RDB를 수직 확장한다", "Redis를 100대로 늘린다" 같은 답은 성립하지 않는다.
- 사용 가능한 캐시 인프라는 **Redis** 로 명시되어 있다.
- **실행 가능성**: 코드 수정 없이 실행 가능해야 한다. 추가 인프라가 필요한 경우 README 명시.

### 0.8 명시되지 않은 정보와 채택된 가정

요구사항은 "분산 환경", "TPS 목표", "Redis / MySQL 사용"까지만 명시한다. 그 외에 **서버 스펙**, **네트워크 토폴로지**, **PG 응답 특성**, **결제 수단별 정책** 등은 명시되어 있지 않다.

이 프로젝트는 이 빈칸을 *명시적 가정*으로 메운다. 가정이 깨지면 결정의 토대도 흔들리므로, 이후의 모든 쟁점은 이 가정을 전제로 평가된다.

| 항목 | 명시 여부 | 이 프로젝트의 채택 가정                                                                 |
|---|---|-------------------------------------------------------------------------------|
| 앱 서버 스펙 | 명시 X | 4 vCPU / 8 GB RAM 기준. 로컬 개발은 단일 앱 프로세스, 분산성 검증은 필요 시 동일 앱 다중 인스턴스 실행으로 확인     |

#### 이 가정이 결정에 끼친 영향

- **Redis와 DB가 동일 데이터센터에 있다는 가정** 위에서 "Redis 게이트 + DB 조건부 UPDATE"의 두 단계 round-trip이 합쳐 < 5 ms로 끝난다고 본다. 만약 RTT가 수십 ms라면 단일 Redis 모델은 통신 비용 때문에 성립하기 어렵다.
- **앱 서버가 4 vCPU / 8 GB 수준이라는 가정** 위에서 1,000 TPS를 받아낸다. 톰캣 스레드 / DB 커넥션 풀 / JVM 힙 / Redis 커넥션 풀은 이 사양에 맞춰 산정된다. 다중 앱 인스턴스 검증이 필요한 경우 같은 애플리케이션을 여러 프로세스로 실행한다. 더 작은 인스턴스라면 백프레셔 임계와 큐 깊이가 달라진다.
- **PG 응답을 p99 1.5 s 로 가정**한 것이 결제 트랜잭션 타임아웃과 점유 토큰 TTL을 결정한다.
- **단일 지역 사용자 가정** 덕분에 도착 순서 ≈ 처리 순서로 두는 것이 사회적으로 공정하다고 본다. 글로벌 사용자라면 *지리적 RTT 차이로 인한 구조적 불공정*을 별도로 다뤄야 한다 (쟁점 2 참조).
- **Redis 단일 인스턴스 선택**은 이 시스템의 *게이트 직렬화* 모델과 정합한다. 이 시스템이 Redis에 의존하는 핵심 보장은 "단일 스레드 명령 실행 = 도착순 직렬화"이며, 이는 단일 인스턴스에서 가장 자연스럽게 발현된다. Redis Cluster는 본질이 *샤딩*이므로 이 모델과 정합하려면 *모든 키를 단일 hashtag*로 묶어 한 슬롯에 강제 배치해야 한다 (즉 Cluster를 *Sentinel처럼 단일 마스터로* 운영). 이 보강은 가용성 향상이 목적이지 처리량 증대가 아니다 (쟁점 5 참조).

이 프로젝트는 위 가정값을 모두 README의 "환경 / 가정" 섹션과 `application.yml`의 주석으로 명시한다.

### 0.9 모호한 요구사항과 채택된 해석

요구사항 명세 일부 항목에서 해석의 여지를 남긴다. 이 프로젝트가 어떤 해석을 택했는지를 명시한다. 해석은 다음 세 가지 기준으로 결정했다.

1. 요구사항 본문의 다른 표현과의 정합성
2. 일반적 결제 시스템 관행
3. 검증 용이성

#### #1의 "시간 기준"에 대한 보충

"동등한 확률"이라는 표현은 엄밀히는 추첨 모델에 가깝지만, 같은 문서에서 "선착순"을 핵심 도메인으로 제시한다. 따라서 이 프로젝트는 모든 사용자가 서버 게이트에 도달한 이후 동일한 처리 규칙을 적용받는다는 의미의 **절차적 공정성**으로 해석한다.

도착순의 "시간"은 *서버 게이트(Redis)에 도달한 시각*을 의미한다. **"현실의 절대 시각(예: 00:00:00.000)"은 기준이 될 수 없다**. 클라이언트의 시계는 신뢰하기 어렵고, 사용자마다 네트워크 RTT도 다르기 때문이다. 따라서 이 시스템의 공정성은 *서버 측 도착 순서*로 정의되며, 그 결과 "네트워크가 빠른 사용자가 유리해진다"는 *구조적 한계*를 인정한다.

이 한계는 이 시스템의 외부 요인으로 본다. 이 시스템이 책임지는 범위는 **"서버 게이트에 도달한 이후의 처리 순서가 깨지지 않는다"**까지이며, 이것이 단일 Redis 게이트로 보장되는지는 쟁점 2에서 별도로 다룬다.

위 해석들은 가능한 한 *보수적*으로 잡았다. 즉 "성공 경로를 좁히고 실패 / 거절 경로를 넓히는" 방향이다. 이는 이 시스템이 *거절을 빠르고 공정하게 수행하는 시스템*이라는 0.2의 관점과 일관된다.

각 해석은 이후 쟁점에서 다시 인용된다. 예를 들어 #1(공정성)은 쟁점 2, #3(멱등키)은 쟁점 3, #4(복합결제 부분 실패)와 #6(결제 실패 분류)은 쟁점 6, #13(Redis 구성)은 쟁점 5에서 다룬다.

## 쟁점 1. 재고 정합성을 어디서, 어떻게 보장할 것인가

### 상황

00시에 한정 10개 상품이 오픈된다. 평시 50 TPS이지만 오픈 직후 1-5분간 500-1,000 TPS의 트래픽 스파이크가 예상된다.
즉 5분 동안 약 30만 건의 요청이 *동일한 10개의 재고*를 두고 경합한다.

다음 두 가지를 동시에 보장해야 한다. Redis 장애처럼 판매 가능 상태가 아닌 경우에는 DB로 우회하지 않고 fail-fast하여 시스템 붕괴와 초과 판매를 막는다.

- **정상 상태의 판매 완결성** — 정상 운영 상태에서는 재고가 남았는데 "재고 없음"이 발생해서는 안 된다.
- **초과 판매 금지** — 어떤 경우에도 11번째 예약이 확정되어선 안 된다.

장애 상태에서는 두 목표의 우선순위가 달라진다. Redis 장애, PG 결과 불명, 보상 실패처럼 안전한 판매 가능 여부를 판정할 수 없는 경우에는 남은 재고가 있더라도 판매를 멈춘다. 이는 미달 판매를 허용한다는 뜻이 아니라, 장애 상황에서 correctness를 availability보다 앞에 두는 보수적 실패 정책이다.

추가 제약 (요구사항 분석 0.7 참조):

- 앱 서버는 2대 이상의 분산 환경이다 → 각 인스턴스의 인메모리 락은 무의미하다.
- 인프라 증설(scale-up/out)이 제한적이다 → "RDB를 수직 확장한다", "Redis Cluster를 100대로 늘린다" 같은 답은 부적절하다.
- Redis는 사용 가능한 인프라로 명시되어 있다.

### 선택지

| 안 | 메커니즘 | 핵심 트레이드오프 |
|---|---|---|
| A | DB 비관적 락 (`SELECT ... FOR UPDATE`) | 가장 단순. 하지만 1,000 TPS가 단일 row 락 대기열에 쌓이며 커넥션 풀 고갈·락 타임아웃 위험. 평시는 OK, 오픈 직후 5분에 가장 먼저 무너짐 |
| B | DB 조건부 UPDATE 단독 (`UPDATE ... WHERE qty > 0`) | 별도 version 조회 없이 한 SQL로 원자적 차감 가능. 그러나 모든 요청이 DB 단일 row로 몰리면 인덱스 래치·버퍼풀 경합으로 p99가 폭발할 수 있음 |
| C | Redis 원자 연산 (`DECR` / Lua) 단독 | 인메모리 단일 스레드 직렬화로 대부분의 요청을 빠르게 거를 수 있음. **단점: SPoF 우려에 더해, "Redis = 진실 / DB = 기록"의 *이기종 데이터베이스 간 정합성*을 직접 합의해야 함** — 운영 난이도가 본질적으로 상승 |
| D | 분산 락 (Redisson / Redlock) | 의미는 명확하지만 락 획득·해제 round-trip이 매 요청마다 추가됨. 1,000 TPS에서 락 큐 자체가 새로운 핫스팟이 됨 |
| E | 메시지 큐(Kafka / Redis Stream)로 직렬화 | FIFO·직렬 처리를 명시적으로 모델링할 수 있음. 그러나 인프라가 늘고 사용자 응답이 비동기가 되어 polling/SSE가 필요. 현재 범위에 비해 과함 |
| F | **Redis 게이트 + DB 재고 선점 (하이브리드)** | Redis Lua로 1차 게이트 → 통과자만 DB 조건부 UPDATE로 재고를 먼저 선점. 코드가 약간 늘지만 양쪽 장애에 강하고, 멱등성·공정성과 자연스럽게 결합 |

**선택: F. Redis 게이트 + DB 재고 선점**

### 왜 그렇게 판단했는가

A·B는 "단순함"이라는 매력이 있다. 특히 B는 낙관적 락(version compare)이 아니라 `qty > 0` 조건만 보는 단일 UPDATE다. version을 먼저 읽고 재시도하는 방식은 단일 row 경합에서 불필요한 충돌을 만들 수 있으므로 이 시스템의 후보에서 제외한다. 그러나 30만 요청이 단일 row 한 곳으로 수렴하는 시나리오에서 RDBMS는 락이든 인덱스 래치든 어떤 형태로든 직렬화 비용을 치른다. 평시에는 동작하지만, 이 시스템의 핵심
시나리오인 *오픈 직후 5분*에 가장 먼저 무너진다. "인프라 증설이 제한적"이라는 조건은 "DB만으로 버틴다"라는 답을 사실상 배제한다.

D는 의미가 명확하지만, 매 요청마다 락 획득/해제 통신이 추가되어 1,000 TPS에서는 락 자체가 또 다른 큐가 된다. 여러 자원에 걸친 작업을 함께 보호할 때는 유용하지만, 단일 카운터 차감에는 비용이 더 큰 선택이다.

E는 가장 견고한 답이지만 인프라(브로커/컨슈머/응답 대기) 비용이 크고, 처음 읽는 사람이 짧은 시간 안에 흐름을 파악하기 어렵다. 이 시스템 규모에서 정당화되지 않는다.

C(Redis 단독)는 빠르지만 두 가지 비용을 함께 진다. **첫째, SPoF**. Redis 인스턴스가 죽으면 *재고 진실* 자체가 사라진다. **둘째, 이기종 데이터베이스 간 정합성**. 이 시스템에서 결제·주문은 결국 DB에 기록되므로 "Redis = 진실 / DB = 기록"이라는 비대칭이 생긴다. 두 시스템이 같은 진실을 공유해야 하는데 한쪽은 인메모리·휘발성, 다른 쪽은 디스크·영속성이다 — 장애 복구 시 *지금 재고는 몇 개인가*에 대한 합의 자체가 어렵다 (Redis 백업으로 결정? DB의 주문 카운트로 역산? 어느 쪽도 정확하지 않다). 이 비대칭을 해소하는 비용이 이 시스템 규모에서 정당화되지 않는다.

#### F를 선택한 이유

1. **DB는 SPoF가 되기 쉬운 자원이다 — 결정된 트래픽만 도달시킨다 (설계 철학).**
   이 시스템에서 정합성의 *최종 보루*는 SQL이지만, 정합성을 *최종 보루에서만* 푼다는 것은 곧 모든 요청을 DB까지 흘려보낸다는 뜻이다. 이는 그대로 DB를 SPoF로 만든다. Redis 게이트는 단순한 "DB 부하 감소 우회로"가 아니라, **DB에 도달해도 의미가 있는 트래픽만 통과시키는 결정 단계**다. 거절될 운명의 요청은 DB까지 가지 않는다 — 이것이 이 시스템 정합성 설계의 출발점이다.

2. **Redis는 게이트, DB는 진실(source of truth)이다.**
   Redis Lua가 30만 요청 중 대부분을 DB 진입 전에 거른다. 살아남은 소수만 DB로 진입하므로 DB는 단일 재고 row에 대한 대규모 경합을 직접 받지 않는다.
   Redis gate 통과는 예약 성공이 아니라 **DB 재고 선점 시도권 획득**이다. 최종 예약 가능 여부는 DB `stock` 조건부 UPDATE의 영향 row 수로 결정한다.

3. **두 단계 모두 원자적이고, 서로의 마지막 방어선이 된다.**
   Redis는 단일 스레드로 직렬화되고, Redis를 통과한 요청만 DB 트랜잭션에 진입한다. DB 트랜잭션은 다음 조건부 UPDATE로 재고를 먼저 선점하고, 같은 트랜잭션 안에서 `PENDING_PAYMENT` 예약을 만든다.

   ```sql
   UPDATE stock SET qty = qty - 1 WHERE product_id = ? AND qty > 0
   ```

   영향 row 0이면 재고 소진, 1이면 선점 성공. 이 한 줄이 *DB 진실의 마지막 보장*이다. 결제는 DB 재고 선점이 성공한 뒤에만 호출한다. 따라서 "결제는 성공했는데 재고가 없다"는 흐름을 구조적으로 피한다.

4. **Redis 카운터는 *트래픽 차단선*이지 재고의 진실이 아니다.**
   Redis가 들고 있는 정수는 "이 시점부터 더 받지 말자"는 *보수적인 게이트*일 뿐이다. 재고의 정확성은 DB의 조건부 UPDATE가 보장한다. 결제 실패가 확정되면 DB stock을 복구하고 Redis
   gate도 `INCR`로 복구한다. 이 보상은 checkoutId 기준 상태 전이로 멱등하게 수행한다. 같은 checkoutId에 대해 이미 `COMPENSATED` 또는 `REFUND_FAILED`로 전이된 보상은 재고를 다시 늘리지 않는다.

   동기화는 다음 두 *이벤트 시점*에만 발생한다.
   - **오픈 직전 1회 초기화** — DB 재고를 Redis에 SET (one-shot, 운영자 트리거).
   - **결제 실패 시 재고 복구** — DB stock 복구 + Redis `INCR`. 이는 checkoutId 기준 멱등 보상 단계이며, Payment 상태 전이를 통해 한 번만 수행한다.

5. **공정성과 자연스럽게 결합된다.**
   단일 Redis는 명령을 한 줄로 직렬화하므로, 서버 게이트에 도달한 이후의 순서를 별도 큐 없이 다룰 수 있다. (쟁점 2에서 별도 논의)

6. **멱등성과 자연스럽게 결합된다.**
   Redis 게이트는 점유 토큰을 발급하고, DB는 `payment.checkout_id UNIQUE` 제약으로 중복 결제를 차단한다 (멱등키 = 주문서 ID). (쟁점 3에서 별도 논의)

#### 결정의 함의 (구현이 따라야 할 것들)

- 재고 차감은 **Redis Lua 스크립트(원자) → 점유 토큰(짧은 TTL) 발급**으로 시작한다.
- Redis gate 통과 후 **DB 트랜잭션에서 재고 선점과 `PENDING_PAYMENT` Booking 생성을 함께 수행**한다.
- 결제는 DB 재고 선점이 성공한 뒤에만 호출한다.
- DB가 "진실"이므로, 운영 중 재고 판단의 기준은 `SELECT qty FROM stock WHERE product_id = ?` 결과다.
- Redis와 DB의 drift는 결제 실패 보상으로 해소한다. PG 실패 시 DB stock 복구와 Redis `INCR`를 함께 시도한다. 보상 실패는 알림 대상이며, 같은 checkoutId로 재실행해도 중복
  복구되지 않도록 Payment 상태 또는 별도 보상 이력으로 "이미 복구됨"을 판정한다.
- **Redis 응답을 받지 못하면 즉시 "예약 불가"로 응답한다** (`503 Service Unavailable` + `Retry-After`). DB로 트래픽을 흘리지 않는다. 이는 임시 fallback이 아니라 *Redis = 게이트*라는 정의의 자연스러운 귀결이다 — 게이트가 없으면 입장도 없다. 자세한 정책과 보강(Redis Cluster의 failover, 앱 단 rate limiter)은 쟁점 5에서 다룬다.

#### 사용 기술의 실제적 보장 근거

- Redis는 **단일 스레드 명령 실행**을 보장한다. Lua 스크립트는 원자적으로 실행되므로 `if qty > 0 then DECR; return token` 조합이 깨지지 않는다.
- MySQL InnoDB는 단일 row UPDATE에 자동 row-lock을 걸고, `WHERE qty > 0` 조건이 거짓이면 영향 row 0을 반환한다 — 이것이 **부정 결과를 호출자가 안전하게 인지**할 수 있게 해 준다.

### 관련 항목 (이후 쟁점에서 이어짐)

- **쟁점 2** — 단일 Redis 인스턴스의 도착순 = 처리순이라는 가정은 어디까지 정당한가
- **쟁점 3** — 점유 토큰 + 멱등키의 결합 방식
- **쟁점 5** — Redis 장애 시 어떤 경로가 fail-fast되고, 어떤 경로가 graceful degrade되는가
- **쟁점 7** — 위 조건부 UPDATE를 JdbcTemplate으로 짜는 것과 JPA로 짜는 것의 차이

## 쟁점 2. 선착순 공정성 — 도착순 보장 방식

### 상황

0.9 #1에서 합의한 공정성의 정의는 *서버 게이트(Redis)에 도달한 시각 기준의 도착순(FIFO)*이다. 이를 분산 환경(앱 서버 2대 이상)에서 어떻게 보장할 것인가? 클라이언트 시계는 신뢰 불가, 앱 인스턴스마다 도착 시각이 다를 수 있다.

### 선택지

| 안 | 메커니즘 | 핵심 트레이드오프 |
|---|---|---|
| A | **단일 Redis 단일 스레드** — 모든 앱 인스턴스가 동일 Redis 인스턴스에 접속, Lua가 도착 순서대로 실행 | 별도 인프라 없이 자연스러운 FIFO. 단일 Redis가 SPoF (쟁점 5에서 별도) |
| B | 대기열(waiting room) — 모든 사용자를 별도 큐에 넣고 순번 부여 | 가장 명시적. 그러나 인프라 복잡, 사용자 경험 변화(폴링/SSE), 이 시스템 규모에 비해 과함 |
| C | 시간 기반 토큰 사전 발급 — 오픈 직전 임의 토큰 N개를 분배 | "공정성"의 의미가 어긋남(누가 미리 받는가가 새 불공정의 원천) |
| D | DB 시퀀스 / AUTO_INCREMENT 기반 순번 부여 | 1,000 TPS 단일 시퀀스 핫스팟 → 쟁점 1에서 사실상 배제된 논리 그대로 |

**선택: A. 단일 Redis 단일 스레드**

### 왜 그렇게 판단했는가

A의 공정성은 Redis의 *실제적 보장*에 의존한다. Redis는 명령을 단일 스레드로 직렬화 실행한다. 따라서 두 앱 인스턴스가 동시에 보낸 명령도 Redis가 받은 순서대로 실행되며, 그 순서를 서버 게이트 기준의 처리 순서로 삼을 수 있다. 별도 큐, 별도 정렬 로직, 별도 시각 동기화가 필요 없다.

B는 *명시적*이지만 비용이 크다. 별도 큐 인프라(Redis Stream, RabbitMQ 등), 사용자 폴링 또는 SSE 채널, 큐의 fail-over까지 — 이 모두를 이 시스템 규모(피크 1,000 TPS, 5분)에서 정당화하기 어렵다. 추가 인프라가 늘면 처음 읽는 사람이 흐름을 파악하는 데 드는 비용도 늘어난다.

C는 "공정성"의 정의 자체와 충돌한다. *누가 사전 발급을 받는가*가 새로운 불공정의 원천이 될 수 있다.

D는 쟁점 1에서 이미 논의된 단일 row 핫스팟 문제를 그대로 반복한다.

#### A의 명시적 한계 — 인정하고 가는 부분

이 시스템은 다음 한계를 *인정한다*. 이를 숨기지 않는다.

1. **네트워크가 빠른 사용자가 유리**. 0.9 #1 보충에서 명시. "현실의 절대 시간"을 기준으로 삼지 않는 한 구조적으로 발생.
2. **봇·자동 스크립트의 우위**. 인간 사용자보다 빠른 클라이언트가 있을 수 있다. 이 보호는 *공정성 메커니즘과 분리*되어 외부 레이트 리밋·캡차 등으로 다룬다 (이 시스템의 책임 범위 외, 쟁점 9에서 메트릭으로 가시화).
3. **Redis Cluster의 failover 윈도우**. 마스터 노드 다운 → replica 승격까지 약 10-30초 동안 게이트가 흔들린다 → 그 시간 동안의 도착 순서는 보장되지 않는다. 이 윈도우 동안 fail-fast로 거절(쟁점 5).

### 결정의 함의

- 모든 앱 인스턴스는 *동일한 Redis 인스턴스*에 접속한다. 인스턴스별로 다른 Redis로 분산하지 않는다.
- 재고 점유 / 멱등성 차단 / 대기 카운터 등 *공정성과 결합된 모든 상태*는 Redis 단일 스레드 실행 경로 안에서 처리된다.
- Redis gate를 통과하지 못한 사용자는 즉시 실패 응답을 받는다. 응답은 정확한 잔여 재고를 노출하지 않고, `sold_out_or_processing`, `retryable`, `retryAfterSeconds`처럼 재시도 가능 여부만 전달한다.
- 결제 실패 보상으로 재고가 복구되면 Redis gate가 다시 열리고, 이후 서버 게이트에 도달한 요청이 그 재고를 획득한다. 별도 대기열이나 푸시 알림은 두지 않는다.
- 단일 인스턴스 Redis의 가용성 보강은 쟁점 5에서.

## 쟁점 3. 멱등성 — 중복 결제 방지

### 상황

주문서에서 짧은 간격(더블 클릭, 네트워크 재시도)에 같은 결제 요청이 여러 번 도달할 수 있다. 0.9 #3에서 합의한 멱등키는 *GET Checkout이 발급한 주문서 ID(checkoutId)*. 클라이언트는 이 ID를 POST Booking 호출 시 그대로 사용한다 — 별도 `Idempotency-Key` 헤더 없이 비즈니스 키 자체로 멱등을 보장한다.

이 모델은 결제 시스템에서 주문·결제 식별자를 재사용하는 일반적인 관행과도 맞다. 클라이언트가 임의의 키를 만드는 것보다 *서버가 발급한 의미 있는 ID*를 재사용하는 편이 추적·디버깅 모두에서 유리하다.

PG사도 보통 주문번호, 결제 요청 키, 멱등키 중 하나를 요구하고 같은 키로 결제 결과를 조회할 수 있게 한다. 따라서 PG 멱등성도 반드시 활용한다. 다만 이 문서의 쟁점은 **PG까지 호출하기 전에 우리 시스템 안에서 중복 요청을 어디까지 차단할 것인가**이다.

### 선택지

| 안 | 메커니즘 | 트레이드오프 |
|---|---|---|
| A | PG 멱등키 단독 — checkoutId를 PG 멱등키로 전달하고 결과 조회 | 외부 결제망의 중복 결제는 막을 수 있음. 그러나 우리 서버의 결제 준비 로직, 재고 점유, 포인트 차감, 응답 재생까지 PG에 위임할 수는 없음 |
| B | DB UNIQUE INDEX 단독 — `payment.checkout_id` UNIQUE | 가장 단순. 그러나 중복 요청이 결제 준비 단계까지 진행된 뒤 DB에서 거절되므로 불필요한 내부 작업이 발생 |
| C | Redis SETNX 단독 — 첫 요청만 키 점유 | 빠르고 이르게 차단. 그러나 Redis 다운 시 차단선 사라짐, 영속성 없음 |
| D | **Redis SETNX 1차 + DB UNIQUE 2차 + PG 멱등키 (하이브리드)** | Redis로 *조기 차단*, DB UNIQUE로 *내부 최종 보장*, PG 멱등키로 *외부 결제망 최종 보장*. 방어선은 늘지만 각 계층의 책임이 명확함 |

**선택: D. Redis SETNX 1차 + DB UNIQUE 2차 + PG 멱등키**

### 왜 그렇게 판단했는가

A(PG 멱등키 단독)는 외부 결제망 관점에서는 타당하다. 같은 checkoutId를 PG 멱등키로 전달하고, 재요청 시 PG의 결제 결과를 조회하면 이중 결제 위험을 줄일 수 있다. 하지만 PG 멱등성은 **PG 바깥의 상태**를 보호하지 못한다. 예를 들어 우리 서버의 재고 점유, 포인트 차감, 주문 생성, 응답 재생 정책은 여전히 별도로 일관성을 맞춰야 한다.

B(DB UNIQUE 단독)는 내부 최종 보장으로는 유용하지만 *늦다*. 중복 요청이 결제 준비 단계까지 진행된 뒤 DB에서 거절되므로 불필요한 내부 작업이 발생하고, 그 사이에 포인트 차감이나 PG 호출 같은 부수효과가 끼어들지 않도록 호출 순서를 매우 조심해야 한다.

C(Redis SETNX 단독)는 *조기 차단*은 되지만 영속성이 없다. Redis 키 유실이나 만료가 발생하면 중복 결제 차단선이 약해진다. SETNX 키가 만료된 후의 동일 키 재요청에 대한 *결과 응답*도 별도로 풀어야 한다.

D는 세 계층의 강점을 결합한다.

1. **Redis SETNX**가 첫 차단선. `SETNX idempotency:{checkoutId} processing EX 24h`. 이미 키가 있으면 "이미 처리 중 또는 처리 완료"로 즉시 응답.
2. **DB UNIQUE INDEX**가 최종 보장. `payment` 테이블에 `checkout_id` UNIQUE. 어떤 race가 있더라도 두 번째 INSERT는 IntegrityViolation으로 실패.
3. **PG 멱등키**가 외부 결제망의 최종 보장. PG 호출 시 checkoutId를 주문번호/멱등키로 전달하고, 응답을 받지 못한 경우 같은 키로 결과를 조회한다.
4. **응답 캐싱**. 처리 완료 시 Redis 키 값에 저장된 응답을 직렬화 보관 (`idempotency:result:{checkoutId}`). 재요청 시 캐시된 응답 그대로 재생.

SETNX는 조기 차단선일 뿐, 검증 실패까지 영구적으로 고정하는 장치가 아니다. Checkout 없음 / 만료 / 사용자 불일치 / 결제 조합 오류처럼 재고·포인트·PG에 부작용을 만들기 전의 실패는
`idempotency:{checkoutId}`를 즉시 삭제한다. 반대로 DB 재고 선점 이후의 실패는 상태와 보상 결과를 남기고, 같은 checkoutId 재요청에는 기존 처리 결과를 재생한다.

### 결정의 함의

- POST Booking 본문은 `checkoutId`를 *필수*로 받는다. 누락 시 `400 Bad Request`.
- checkoutId 형식: UUIDv4. GET Checkout 응답에 포함되어 발급된다.
- checkoutId의 수명 = 결제 윈도우 (24시간). 만료 시 새 GET Checkout 호출 필요.
- PG 호출 시 checkoutId를 PG 주문번호 또는 멱등키로 전달한다. PG 응답 미수신 시 같은 키로 결제 결과를 조회한다.
- Redis 키: `idempotency:{checkoutId}`. 값: `processing` → `done`.
- 응답 재생 시점: Redis 키가 `done` 상태면 저장된 응답 재생. `processing` 상태면 `409 Conflict` (또는 짧은 대기 후 재시도 안내).
- DB UNIQUE 충돌 시 (Redis 키 만료·동시성 경합 등): 저장된 응답 재생 시도, 없으면 처리 결과 조회 후 응답.
- SETNX 이후 검증 실패 시 처리: 재고 게이트에 들어가기 전의 순수 검증 실패는 `idempotency:{checkoutId}`를 삭제해 checkoutId가 `processing`에 갇히지 않게 한다.

#### 사용자별 고립의 자연스러운 해결

checkoutId는 GET Checkout 호출 시 user_id와 함께 서버에서 발급되므로 *사용자별로 자연스럽게 고립*된다. 다른 사용자가 우연히 같은 UUID를 *알아내도* 그 checkoutId의 발급 사용자가 아니면 POST Booking에서 사용자 불일치로 거절된다 (0.9 #10의 `X-User-Id` 가정과 결합).

#### GET Checkout 자체의 멱등성은 — *의도적으로 비멱등*

같은 사용자가 GET Checkout을 여러 번 호출하면 매번 새 checkoutId가 발급된다. 이는 *의도된 설계*다.

- 사용자가 결제 절차 도중 *다시 시작*하고 싶을 때 자연스럽게 새 주문서를 받는다.
- 같은 checkoutId 안에서만 멱등성이 보장된다 (한 주문서 = 한 결제 시도).

## 쟁점 4. 결제 수단 확장성

### 상황

지원 결제 수단: 신용카드 / Y페이 / Y포인트. 허용 조합: 단독, (카드 + 포인트), (Y페이 + 포인트). 금지: 카드 + Y페이. **신규 결제 수단 추가 시 Booking API의 비즈니스 로직 수정 최소화**가 요구사항의 명시적 요구.

### 선택지

| 안 | 메커니즘 | 트레이드오프 |
|---|---|---|
| A | if-else / switch 분기 | 가장 단순. 신규 수단 추가 시 모든 분기 갱신 → 확장 비용 큼 |
| B | Strategy 패턴 (`PaymentMethod` 인터페이스) 단독 | 단일 수단은 깔끔. 그러나 *복합결제 조합 검증*과 *순차 실행/보상*을 표현할 수 없음 |
| C | Chain of Responsibility | 순차 실행은 자연스러움. 그러나 조합 검증(예: 카드 + Y페이 금지)이 체인 외부에 분산되면서 응집도 ↓ |
| D | **Strategy + Composer + Validator (계층 분리)** | Strategy = 개별 수단, Composer = 순차 실행/보상, Validator = 조합 규칙. 신규 수단 = Bean 추가 + Validator 한 줄 |

**선택: D. Strategy + Composer + Validator**

### 왜 그렇게 판단했는가

이 도메인은 세 가지 *서로 다른 책임*이 명확히 구분된다.

1. **개별 수단의 동작** — 카드는 PG 호출, 포인트는 사용자 잔액 차감, Y페이는 별도 PG.
2. **여러 수단의 조립** — 순차 실행, 부분 실패 시 보상. 포인트 차감 → 카드 결제. 카드 실패 시 포인트 INCR.
3. **조합 규칙** — 카드 + Y페이 금지, 합계 일치.

이 세 책임을 한 클래스에 몰면 if-else가 빠르게 늘어난다. 셋을 *타입*으로 분리하면 신규 수단을 추가할 때 각 책임의 변경 지점이 명확해진다.

#### 신규 수단 추가 시뮬레이션 — "선물카드"가 추가된다면

```
1. GiftCardPayment implements PaymentMethod  (Bean 추가)
2. PaymentValidator 규칙 한 줄 추가         (예: "선물카드는 단독만 허용")
3. (필요 시) Composer의 실행 순서 명시       (예: "선물카드는 카드/페이보다 먼저")
```

`BookingService`는 수정하지 않는다. 이것이 요구사항의 본질이다.

### 결정의 함의 (코드 골격)

```java
// 1. 개별 수단 (Strategy)
interface PaymentMethod {
    PaymentMethodType type();
    PaymentResult charge(UserId userId, Money amount, PaymentContext ctx);
    void refund(PaymentResult prior);
}

class CardPayment  implements PaymentMethod { ... }
class YPayPayment  implements PaymentMethod { ... }
class PointPayment implements PaymentMethod { ... }

// 2. 조립 + 보상 (Composer)
class PaymentComposer {
    private final PaymentValidator validator;
    private final Map<PaymentMethodType, PaymentMethod> methods;

    PaymentResult compose(List<PaymentRequest> requests) {
        validator.validate(requests);  // 조합 규칙 검증
        List<PaymentResult> succeeded = new ArrayList<>();
        for (PaymentRequest req : requests) {
            try {
                succeeded.add(methods.get(req.type()).charge(...));
            } catch (PaymentException e) {
                // 보상 — 이미 성공한 것들을 역순으로 환불
                Lists.reverse(succeeded).forEach(r -> methods.get(r.type()).refund(r));
                throw e;
            }
        }
        return PaymentResult.combine(succeeded);
    }
}

// 3. 조합 규칙 (Validator)
class PaymentValidator {
    void validate(List<PaymentRequest> requests) {
        boolean hasCard = requests.stream().anyMatch(r -> r.type() == CARD);
        boolean hasYPay = requests.stream().anyMatch(r -> r.type() == YPAY);
        if (hasCard && hasYPay) throw new InvalidCompositionException("카드 + Y페이 금지");
        // 합계 일치 검증, 한도 초과 검증 등
    }
}
```

`BookingService`는 `PaymentComposer.compose(...)`만 호출한다. 결제 수단 도메인의 변화는 Booking 로직으로 새어 나오지 않는다.

#### JPA로 했다면? (쟁점 7과 동일 정신)

이 패턴은 영속성 기술과 무관하므로 JPA에서도 동일한 클래스 구조로 표현된다. 차이는 `refund` 시 영속성 컨텍스트의 1차 캐시가 *다른 트랜잭션의 변경*을 보지 못해 추가 `refresh` 또는 `flush+clear`가 필요할 수 있다는 점뿐.

## 쟁점 5. Redis 구성 및 장애 대응

### 상황

이 시스템에서 Redis는 *재고 / 멱등성 / 공정성의 게이트*다. 요구사항은 Redis 사용을 요구하지만, 단일 인스턴스 / Sentinel / Cluster 중 어떤 구성을 쓰라는지는 명시하지 않는다.

따라서 이 쟁점에서는 먼저 Redis 구성을 선택하고, 그 선택 위에서 장애 시 Fallback 전략을 정한다.

### 구성 선택지

| 안 | 구성 | 트레이드오프 |
|---|---|---|
| A | **단일 Redis 인스턴스** | 구조가 가장 단순하고, 단일 스레드 실행 순서를 그대로 게이트 순서로 사용할 수 있음. 단점은 Redis 자체가 SPoF가 된다는 점 |
| B | Redis Sentinel | 단일 마스터 모델을 유지하면서 failover 가능. 다만 Sentinel 노드와 운영 절차가 추가됨 |
| C | Redis Cluster | failover와 샤딩을 함께 제공. 다만 이 시스템의 핵심 키를 여러 슬롯에 나누면 게이트 순서가 깨질 수 있어, 결국 단일 hashtag로 한 슬롯에 묶어야 함 |

**선택: A. 단일 Redis 인스턴스**

### 왜 단일 인스턴스를 선택했는가

이 시스템이 Redis에 기대는 핵심 가치는 처리량 확장이 아니라 **게이트의 직렬화**다. 모든 앱 인스턴스가 하나의 Redis에 Lua 스크립트를 실행하면, Redis가 받은 명령 순서를 서버 게이트 기준의 처리 순서로 삼을 수 있다. 재고가 10개뿐인 이벤트에서 Redis를 샤딩해 처리량을 늘리는 것보다, 하나의 결정 지점을 두는 편이 공정성 설명과 구현 검증이 단순하다.

Redis Cluster는 처리량 확장 관점에서는 매력적이지만, 이 시스템에서는 이점이 제한적이다. 재고 / 멱등성 / 점유 토큰처럼 공정성과 결합된 키가 여러 슬롯으로 흩어지면 순서 보장이 약해진다. 이를 피하려면 모든 관련 키를 같은 hashtag로 묶어 단일 슬롯에 넣어야 하는데, 이 경우 Cluster는 사실상 "가용성 보강용 단일 마스터"처럼 쓰인다. 즉 처리량 확장 효과는 크지 않고 운영 복잡도만 늘어난다.

따라서 기본 구현은 단일 Redis 인스턴스로 둔다. 대신 이 선택의 약점인 Redis 장애는 숨기지 않고, 아래처럼 fail-fast와 선택적 failover 보강으로 다룬다.

### 장애 대응 선택지

| 안 | 정책 | 트레이드오프 |
|---|---|---|
| A | **Fail-fast** — Redis 응답 없으면 즉시 `503 + Retry-After`. DB로 흘리지 않음 | 가장 정직함. 부분 서비스 중단을 수용하는 대신 DB를 보호 |
| B | DB-only fallback — Redis 다운 시 DB 조건부 UPDATE만으로 처리 | 1,000 TPS 단일 row 핫스팟 직격 → *쟁점 1에서 사실상 배제된 논리* |
| C | A + Redis Cluster failover | failover 윈도우(약 10-30초) 외엔 정상 동작. 인프라 5-6대 추가 (Cluster 최소 구성) |
| D | A + Redis Cluster + 앱 단 in-process 토큰 버킷 | failover 윈도우 동안에도 *극도로 보수적*인 통과 허용. 코드 +α |

**선택: A를 핵심 정책으로, C·D를 보강 옵션으로 둔다**

이 프로젝트의 구현 범위는 다음과 같다.

- **A: 필수 구현** — Redis 클라이언트 timeout 짧게(200 ms) + Resilience4j Circuit Breaker + 503 응답
- **C: 설계 옵션** — Redis Cluster / Sentinel은 가용성 보강 선택지로만 둔다. 본 프로젝트의 기본 구성은 단일 Redis 인스턴스다
- **D: 코드 옵션** — 앱 단 토큰 버킷(예: Bucket4j) 인터페이스 명시, 필요 시 활성화

### 왜 그렇게 판단했는가

이 시스템의 "Fallback 전략"은 *fallback을 제공하지 않는다*가 아니라 **"의도적 fail-fast"**라고 표현하는 것이 정직하다. 요구사항을 부정하지 않으면서, Redis 장애 시 어떤 가치를 우선할지 명확히 밝힌다.

B는 쟁점 1에서 이미 배제한 논리를 다시 적용하는 선택이다. Redis가 죽었다고 해서 1,000 TPS 단일 row 핫스팟을 DB가 갑자기 견딜 수 있게 되지는 않는다. 오히려 재고는 안전할 수 있어도 DB까지 같이 무너져 다른 거래(정산, 마감, 다른 상품 조회)까지 영향을 받을 수 있다.

A의 단순함은 이 시스템의 *Redis = 게이트* 정의에서 나온다. 게이트가 닫혔으면 입장도 허용하지 않는다. 이 단순함이 강점이다.

C는 *failover 윈도우*를 줄이는 데 효과적이다. 마스터 노드 다운 → replica 승격까지의 시간을 수 초-수십 초 수준으로 줄일 수 있다. 인프라 비용은 작은 인스턴스 5-6대 추가 (Cluster 최소 구성: 3 master + 3 replica)이다.

> **Sentinel을 쓰지 않는 이유**. Sentinel(별도 감시 노드 + 단일 마스터 + replica)도 비슷한 효과를 내지만 *독립 컴포넌트가 한 종류 더 늘어난다*. Cluster는 자기 자신이 failover의 결정 주체이므로 운영 컴포넌트가 줄어든다. 다만 Cluster의 샤딩이 이 시스템 모델과 충돌하지 않도록 *모든 키를 단일 hashtag*로 묶어 단일 슬롯에 보관한다. 이 시스템은 샤딩을 통한 처리량 증가보다 게이트 직렬화가 핵심이다.

D는 *failover 윈도우 안에서도* 일부 통과를 허용한다. 예를 들어 Cluster가 failover하는 동안 모든 사용자를 거절하지 않고, 인스턴스당 5 RPS 정도만 받아들이는 방식이다. 이 통과량은 DB가 충분히 견딜 수 있는 수준으로 잡는다. 완전한 fail-fast보다 사용자 경험은 부드럽지만, 구현과 검증 범위가 늘어난다.

### 결정의 함의

- **응답 규약**: Redis 응답 실패 시 즉시 `503 Service Unavailable`. 본문에 사용자 메시지("일시적으로 예약이 폭주하고 있습니다. 잠시 후 다시 시도해 주세요"). 헤더 `Retry-After: 5`.
- **DB 보호**: 어떤 경우에도 Redis 게이트를 우회한 DB 직접 차감은 발생하지 않는다.
- **결제 진행 중 Redis 다운**: 이미 DB 재고 선점까지 완료된 요청은 결제 결과 확정까지 진행한다. 결제 실패 시 DB stock 복구를 먼저 수행하고 Redis `INCR` 복구를 시도한다. Redis 복구 실패는 알림 대상이며, DB로 우회 판매하지 않는다.
- **운영 가시성**: Redis 응답 실패는 알림 메트릭(쟁점 9). 503 응답률 임계 초과 시 운영자 호출.

## 쟁점 6. 결제 실패 처리

### 상황

0.9 #6에서 5종 분류 채택 — 한도 초과, 카드 거절, 잔액 부족, PG 타임아웃, PG 응답 미수신. 각 케이스별 *보상*과 *재시도* 정책이 다르다.

### 선택지

| 안 | 표현 | 트레이드오프 |
|---|---|---|
| A | if-else / switch | 단순하지만 분류 추가/규칙 변경에 취약 |
| B | **Failure Type → Policy 매핑 (enum + 정책 객체)** | 분류와 정책을 분리. 신규 분류 추가 시 enum + 정책 한 줄 |
| C | 외부 룰 엔진 (Drools 등) | 이 시스템 규모에 비해 과함 |

**선택: B**

### 결정의 함의 — 분류 매트릭스

| 분류 | 자동 보상 | 재시도 | 사용자 응답 | 운영자 알림 |
|---|---|---|---|---|
| **한도 초과** | DB stock 복구, Redis gate 복구, 차감된 포인트가 있으면 포인트 복구 | 권장하지 않음 (사용자 한도 변경 필요) | "한도 초과로 결제할 수 없습니다." | 없음 |
| **카드 거절** | DB stock 복구, Redis gate 복구, 차감된 포인트가 있으면 포인트 복구 | 다른 결제수단 권장 | "카드사가 결제를 거절했습니다." | 없음 |
| **잔액 부족** (포인트) | PG 호출 전 실패. DB stock 복구, Redis gate 복구 | 충전 후 재시도 권장 | "포인트가 부족합니다." | 없음 |
| **PG 타임아웃** | *대기* — 즉시 보상하지 않음. checkoutId로 PG 결과 조회 후 결정 | 자동 결과 조회 재시도(최대 N회 / N분) | "결제 결과 확인 중..." | 조회 한도 초과 또는 PG 장애 지속 시 알림 |
| **PG 응답 미수신** | *대기* — 즉시 보상하지 않음. checkoutId로 PG 결과 조회 후 결정 | 자동 결과 조회 재시도(최대 N회 / N분) | "결제 결과 확인 중..." | 조회 한도 초과 또는 PG 장애 지속 시 알림 |

### 왜 그렇게 판단했는가

핵심 원칙 두 가지.

1. **확정 실패는 즉시 보상**. 한도 초과 / 카드 거절 / 잔액 부족처럼 PG가 "확정적으로 실패"라고 응답한 케이스는 즉시 보상(DB stock 복구, Redis gate 복구, 필요한 경우 포인트 복구)이 안전하다. 이중 결제 위험이 없기 때문이다.

2. **결과 불명 상태는 멱등키 기반 조회로 자동 수습한다**. PG 타임아웃이나 응답 미수신은 *결제가 됐는지 안 됐는지 모르는 상태*다. 이때 같은 checkoutId로 PG 결제를 다시 생성하지 않고, 먼저 같은 키로 결제 결과를 조회한다. 조회 결과가 성공이면 예약을 확정하고, 실패면 토큰과 포인트를 보상한다. 조회 한도를 넘기거나 PG 장애가 지속되는 경우에만 운영자 알림을 보낸다.

#### 복합결제 부분 실패 — 0.9 #4와의 결합

복합결제(예: 포인트 + 카드) 중 한 단계가 실패하면 0.9 #4 채택 해석에 따라 자동 보상한다. 결제 실패가 확정되면 결제 수단 보상과 함께 DB stock, Redis gate도 checkoutId 기준으로 멱등 복구한다.

- 포인트 차감 → 카드 결제 실패 → 포인트 복구 + DB stock 복구 + Redis gate 복구. 사용자에게는 "전체 실패" 응답.
- 단, 카드 결제가 결과 불명 상태인 경우 포인트 보상도 보류한다. checkoutId 기반 결과 조회로 결제 결과를 확정한 뒤 일괄 처리한다.

결제 수단별 보상은 `PaymentComposer.compose()`(쟁점 4) 내부의 try-catch가 책임진다. DB stock과 Redis gate 복구는 Booking 흐름의 보상 단계에서 checkoutId 기준으로 멱등 수행한다.

## 쟁점 7. 영속성 선택 — JdbcTemplate vs JPA

### 상황

핵심 SQL — `UPDATE stock SET qty = qty - 1 WHERE qty > 0` — 이 이 시스템 정합성의 *최종 보루*다. 영속성 추상화의 두께가 *동시성 디버깅 가능성*에 영향을 준다.

### 선택지

| 안 | 트레이드오프 |
|---|---|
| A | **Spring JdbcTemplate** — SQL이 그대로 보임. 매핑은 RowMapper |
| B | Spring Data JPA + Hibernate — 도메인 모델 풍부, 1차 캐시 / dirty checking |
| C | JOOQ — 타입 안전 SQL DSL. 코드 생성 |
| D | MyBatis — XML / 어노테이션 SQL |

**선택: A. JdbcTemplate**

### 왜 그렇게 판단했는가

#### 1. 정합성의 최종 보루는 SQL이다

이 시스템의 정합성을 보장하는 핵심은 다음 한 줄이다.

```sql
UPDATE stock SET qty = qty - 1 WHERE product_id = ? AND qty > 0
```

이 SQL이 *코드 위에 그대로 보이는 것*이 이 시스템에서 가장 중요한 가독성이다. JdbcTemplate은 SQL을 숨기지 않는다.

#### 2. JPA의 추상화는 동시성 디버깅을 어렵게 한다

JPA의 영속성 컨텍스트(1차 캐시)와 dirty checking은 단일 트랜잭션 안에서는 매우 편리하지만, *분산 환경에서 다른 트랜잭션이 같은 row를 변경한 직후*에는 *내 1차 캐시가 stale한지*를 의식적으로 신경 써야 한다. 이 시스템의 핵심 시나리오인 "동시 1,000 TPS의 단일 row 차감"에서 이 추상화는 *예상치 못한 동작의 원천*이 된다.

또한 위 핵심 SQL을 JPA로 표현하려면 `@Modifying @Query("UPDATE Stock s SET s.qty = s.qty - 1 ...")` 같은 native-like JPQL을 쓰게 되어 *이미 SQL로 회귀*한다. JPA의 강점이 발휘되지 않는 영역에서 JPA를 쓰는 셈.

#### 3. 이 시스템 도메인은 단순하다

상품 / 재고 / 주문 / 결제 / 멱등키 — 모두 평탄한 테이블이다. 풍부한 객체 그래프, 양방향 연관관계, 컬렉션 lazy loading 같은 JPA 강점이 발휘될 영역이 없다.

#### 4. 학습 친화성

이 프로젝트는 학습 친화성이 가독성과 검증 용이성에 직결된다. SQL이 보이는 코드는 처음 읽는 사람이 *지금 무슨 일이 일어나는지*를 즉시 파악할 수 있다.

### JPA로 했다면 — 비교 코드

쟁점 4의 코드 골격은 영속성 무관하므로 그대로다. 차이는 Repository 계층에서만 발생한다.

```java
// JdbcTemplate (선택)
class StockJdbcRepository {
    int decrease(long productId) {
        return jdbcTemplate.update(
            "UPDATE stock SET qty = qty - 1 WHERE product_id = ? AND qty > 0",
            productId
        );
    }
}

// JPA (비교 — 동일 의미)
interface StockJpaRepository extends JpaRepository<Stock, Long> {
    @Modifying
    @Query("UPDATE Stock s SET s.qty = s.qty - 1 WHERE s.id = :id AND s.qty > 0")
    int decrease(@Param("id") long id);
}
```

코드 양은 비슷하다. 차이는 *호출자가 의식해야 할 것*이 JPA 쪽에서 더 많다는 점이다. 예를 들어 `@Modifying` 호출 후 영속성 컨텍스트 flush/clear, 같은 트랜잭션 안에서 stock 엔티티 재조회 시 stale 가능성을 고려해야 한다.

### 결정의 함의

- 메인 영속성: JdbcTemplate.
- DDL 관리: `schema.sql`.
- JPA 비교 코드는 문서상 예시로만 둔다. 기본 빌드와 런타임 의존성에는 JPA를 포함하지 않는다.
- 트랜잭션: `@Transactional` 그대로 사용. 둘 다 동일.

## 쟁점 8. 인프라 구성

### 상황

요구: *코드 수정 없이 실행 가능*. Redis와 MySQL이 필요한 애플리케이션을 로컬에서 재현 가능한 환경으로 실행해야 한다. 0.7-0.8의 채택 가정은 앱, MySQL, Redis가 같은 개발 머신에서 뜨는 기본 실행 경로와, 필요 시 별도 부하 검증 경로로 나눈다.

### 선택지 — 로컬

| 안 | 트레이드오프 |
|---|---|
| A | **Spring Boot Docker Compose + `./gradlew bootRun`** | 앱 실행 시 MySQL / Redis가 자동 기동. IntelliJ / Gradle 개발 흐름에 가장 편함 |
| B | **수동 `docker compose up` 후 `./gradlew bootRun`** | Docker 인프라를 명령어로 직접 확인하고 싶은 사용자에게 명시적 |
| C | 애플리케이션까지 compose로 패키징 | 운영 형태와 비슷하지만 초기 구현과 디버깅 비용이 늘어남 |

**선택: A와 B를 모두 지원한다. 기본 안내는 A, 명령어 기반 대안은 B로 둔다**

### 선택지 — 배포 / 부하 검증

| 안 | 트레이드오프 |
|---|---|
| A | 클라우드 매니지드 서비스 | 매니지드 기능은 풍부하지만 비용·셋업 시간이 큼 |
| B | **로컬 개발 실행 + k6 별도 실행** | 기본 검증 흐름에 적합. 실제 분산 물리 환경은 아님 |
| C | App / Redis / MySQL / k6 분리 배포 | 부하 검증에는 더 적합하지만 구성 복잡도가 증가 |

**선택: B를 기본 검증 경로로, C를 부하 검증용 옵션으로 둔다**

### 왜 그렇게 판단했는가

- **로컬**: `docker-compose.yml`은 MySQL / Redis 정의의 단일 원천으로 둔다. Spring Boot Docker Compose를 쓰면 `./gradlew bootRun`만으로 이 파일을 읽어 인프라를 자동 기동할 수 있고, CLI 사용자는 같은 파일로 `docker compose up`을 직접 실행할 수 있다.
- **분산 가정**: 애플리케이션은 인스턴스 로컬 메모리에 재고 / 멱등 상태를 두지 않는다. 따라서 분산성의 핵심은 "앱 프로세스 수"보다 "공유 Redis와 DB만을 상태 진실로 쓰는가"다. 필요 시 같은 앱을 다중 포트로 띄워 부하 검증을 추가한다.
- **부하 검증**: 더 정확한 측정이 필요하면 App / Redis / MySQL / k6를 별도 노드로 분리한다. 이는 README의 선택 실행 경로로 둔다.

### 로컬 개발 구성 (요지)

```kotlin
dependencies {
    testAndDevelopmentOnly("org.springframework.boot:spring-boot-docker-compose")
}
```

- `docker-compose.yml`에는 MySQL과 Redis를 둔다.
- 기본 실행은 `./gradlew bootRun`이다. Spring Boot Docker Compose가 `docker-compose.yml`을 감지해 필요한 인프라를 자동 기동한다.
- 수동 실행을 선호하면 `docker compose up -d`로 MySQL / Redis를 먼저 띄운 뒤 `./gradlew bootRun`을 실행한다.
- DB 스키마와 초기 데이터는 `schema.sql`로 관리한다.
- k6 / Prometheus / Grafana는 선택 검증 도구로 분리한다.

### 결정의 함의

- 기본 결과물은 로컬 재현 가능성을 우선한다.
- 외부 배포 URL이나 분리 배포 결과는 실제로 준비한 경우에만 README에 명시한다.

## 쟁점 9. 관측 / 부하 환경

### 상황

이 시스템의 가장 중요한 검증 항목은 *1,000 TPS 시나리오에서 정합성·공정성·멱등성이 정말 깨지지 않는가*이다. 이를 어떻게 *재현 가능한 형태*로 만들 것인가? 실행 환경을 새로 받은 사람도 직접 검증할 수 있어야 한다.

### 결정 — 부하

부하 도구: **k6** (코드형 시나리오). 기본 검증은 k6 stdout 요약을 사용한다. Prometheus remote-write 출력은 선택 확장으로 둔다.

검증 시나리오는 최소 정합성 / 멱등성 중심으로 제공하고, 스파이크 / Redis 장애 시나리오는 선택 확장으로 둔다.

| 시나리오 | 목적 | 통과 기준 |
|---|---|---|
| **A. 정합성** | 정상 운영 상태에서 다수 사용자 동시 진입, 정확히 10건만 성공 | 성공 = 10. 초과 없음. 장애 시나리오는 fail-fast 별도 검증 |
| **B. 멱등성** | 같은 `checkoutId`로 여러 번 동시 결제 호출 | 결제 1건만, 응답 동일 |
| **C. 스파이크** (선택) | 50 TPS → 피크 1,000 TPS | p99와 실패 응답률 확인 |
| **D. Redis 장애** (선택) | 부하 중 Redis 중단 | 503 즉시 발생, DB 우회 없음 |

각 시나리오의 제공 여부와 실행 명령은 README에 명시한다.

### 결정 — 관측

- **메트릭**: Spring Boot Actuator + Micrometer.
    - 필수 노출 (이번 범위에서 구현):
        - HTTP timing / status metrics (Actuator 기본)
        - Redis gate 성공 / 실패
        - DB 재고 선점 성공 / 실패
        - Booking 성공 수
        - 결제 실패 분류
        - 503 응답
        - compensation refund_failed
        - Resilience4j circuit breaker metrics
    - 확장 옵션 (이번 범위에서는 미노출):
        - PG 응답시간 — Fake PG 기반 최소 구현이라 의미 있는 계측이 어렵다. 실제 PG 연동 시 추가한다.
        - 운영자용 잔여 재고 — 사용자 API에는 절대 노출하지 않는다는 원칙과 분리하기 위해 운영자용 내부 지표나 별도 조회 경로로만 제공할 수 있다. 이번 범위에서는 제외한다.
- **시각화**: Grafana 대시보드는 선택 확장이다. 기본 검증은 k6 stdout 요약과 Actuator/Micrometer 메트릭 노출로 충분하다. 제공하는 경우 README에 import 방법을 명시한다.
- **로그**: 표준 stdout. Loki는 옵션(이 프로젝트 범위 외).
- **트레이싱**: 이 프로젝트 범위 외. 결제 흐름 시퀀스는 코드 / 시퀀스 다이어그램으로 대체.

### 왜 그렇게 판단했는가

- k6는 코드형 시나리오라 README에 동봉하기 쉽다. JMeter는 GUI/XML, Gatling은 Scala. k6의 JS는 진입 장벽이 낮고 가이드 친화적.
- Prometheus + Grafana는 사실상 표준에 가깝다. 다만 기본 실행 경로에는 포함하지 않고, 제공하는 경우 선택 관측 환경으로 분리한다. Datadog/New Relic은 이 프로젝트 범위에서 과하다.
- *정합성 / 공정성 / 멱등성*은 모두 부하 시나리오로 *기계적으로 검증*되어야 한다. "정합성을 보장한다"는 *주장*이 아니라 *재현 가능한 숫자*가 검증의 근거가 되어야 한다.

### 결정의 함의

- README에 최소 정합성 / 멱등성 검증 방법을 포함한다.
- 부하는 *별도 프로세스 또는 별도 노드에서* 돌리는 것이 정확하다. 같은 머신에서 k6와 앱을 함께 실행하면 측정에 영향을 줄 수 있으므로 README에 이 한계를 명시한다.
