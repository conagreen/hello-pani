# Skill: task-executor

`docs/TASKS.md`의 특정 Task를 구현할 때 사용한다.
목표는 한 번에 하나의 Task만 끝내고, 완료 조건을 검증 가능한 상태로 남기는 것이다.

## 입력

- 사용자가 지정한 Task 번호 또는 제목
- 기준 문서:
    - `docs/TASKS.md`
    - `docs/DECISIONS.md`
    - `docs/DOMAIN.md`

## 절차

1. `docs/TASKS.md`에서 대상 Task를 찾는다.
2. 대상 Task의 다음 항목을 작업 메모에 요약한다.
    - 목표
    - 입력 문서
    - 작업
    - 완료 조건
    - 설계 불변식
3. 입력 문서를 읽고 해당 Task와 직접 관련된 설계만 추린다.
4. 현재 저장소 상태를 확인한다.
    - `git status --short`
    - `rg --files`
    - 관련 패키지 / 설정 파일 확인
5. 이미 완료된 항목과 새로 해야 할 항목을 구분한다.
6. 구현 범위가 Task 밖으로 커지면 멈추고 사용자에게 범위 확장을 확인한다.
7. 필요한 파일만 수정한다.
8. 테스트 또는 검증 명령을 실행한다. 결정론적으로 끝나는 명령은 직접 실행한다.
9. `docs/TASKS.md`의 체크박스는 사용자가 명시적으로 원할 때만 수정한다.
10. 최종 응답에 다음을 포함한다.
    - 구현 요약
    - 변경 파일
    - 실행한 검증
    - 실패 또는 미검증 항목
    - 다음 Task로 넘어가기 전 주의점

## 구현 불변식

- `schema.sql`을 사용한다. 별도 DB 마이그레이션 도구를 추가하지 않는다.
- `docker-compose.yml`은 MySQL / Redis 정의의 단일 원천이다.
- `developmentOnly("org.springframework.boot:spring-boot-docker-compose")`를 사용한다.
- 실제 PG SDK를 붙이지 않는다.
- Redis 장애 시 DB 우회 fallback을 만들지 않는다.
- 사용자 응답에 잔여 재고 수량을 넣지 않는다.
- Redis gate 통과를 예약 성공으로 처리하지 않는다. 최종 예약 가능 여부는 DB 재고 선점 성공으로 판단한다.
- 외부 PG 호출은 DB 트랜잭션 밖에서 수행한다.
- 결제 실패 보상은 checkoutId 기준으로 멱등하게 구현하고, 단계별 보상 중 실패한 단계만 재시도한다.

## 완료 판단

Task는 다음을 만족할 때 완료로 본다.

- Task의 작업 항목이 구현되어 있다.
- 완료 조건에 해당하는 테스트 또는 검증이 실행되었다.
- 설계 불변식이 깨지지 않았다.
- 관련 문서와 구현이 충돌하지 않는다.
- 미검증 항목이 있다면 이유가 명확히 기록되었다.

## 검증 명령 기준

직접 실행해야 하는 명령:

- `./gradlew test`
- 대상 범위가 좁은 특정 Gradle 테스트
- `rg` 기반 문서 / 코드 검색
- `git status --short`

조건부로 실행하는 명령:

- `./gradlew bootRun`: 실행 가능성 확인이 Task 완료 조건이거나 사용자가 요청한 경우 실행한다. 확인 후 종료한다.
- `docker compose up -d`: 수동 인프라 경로 확인이 Task 완료 조건이거나 사용자가 요청한 경우 실행한다.

주의:

- 명령 실패를 최종 상태로 두지 말고, 수정 가능한 실패는 고친 뒤 재실행한다.
- 실행하지 않은 검증을 실행한 것처럼 쓰지 않는다.
